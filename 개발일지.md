# 📝 개발일지 — AI 사진 분류기

> 사진을 올리고 단어(예: "고양이")를 입력하면 브라우저 안에서 AI가 자동 분류하는 웹앱.
> 작성일: 2026-06-23

---

## 1. 목표

- 사진 여러 장(예: 50장)을 업로드하고, 찾을 단어를 입력하면 해당 사진을 자동으로 모아줌
- **서버 비용 0**, **사진 외부 전송 없음**(프라이버시), 빌드 단계 없이 더블클릭/로컬서버로 실행

---

## 2. 사전 조사 (시장/유사 서비스)

| 유형 | 예시 | 가격 |
|------|------|------|
| 완성형 데스크톱 | Excire Foto, ON1 | $30~129 일회성 |
| 클라우드 | Google Photos, OneDrive | 저장공간 요금 |
| 개발자 API | Imagga($14~/월), Clarifai($30~/월), Google Vision | 종량/구독 |
| 오픈소스(브라우저) | **visheratin/ai-photo-sorter** | 무료 |

**핵심 인사이트**
1. "임의 단어 검색"은 이미 표준 기능(CLIP 기반 제로샷)
2. 차별점은 **프라이버시**(기기 내 처리)와 **가벼운 무료 웹**(시장 빈틈)
3. 브라우저에서 추론하면 사용자가 늘어도 **서버 비용 0**

---

## 3. 참고 구조 분석 — ai-photo-sorter

- **방식**: 브라우저에서 CLIP(ONNX) 직접 실행 → 텍스트/이미지를 같은 공간 벡터로 변환 후 코사인 유사도 비교
- **스택**: `@visheratin/web-ai`(ONNX 런타임) + `onnxruntime-web` + tokenizers + Next.js
- **영리한 점**: Web Worker 오프로딩, IndexedDB 캐싱, PWA(오프라인)
- **유일한 약점**: 첫 로딩에 모델 수백 MB 다운로드 (fp32 CLIP ≈ 577MB)
- ⚠️ `web-ai`는 2026-01 아카이브(개발 중단) → 최신 라이브러리로 교체 필요

---

## 4. 약점 해결 전략 (첫 로딩 용량)

| 방법 | 효과 |
|------|------|
| q8 양자화 | 577MB → ~150MB (¼) |
| 경량 모델(MobileCLIP) | 수십 MB |
| **Transformers.js v3** | 위 둘을 코드 한 줄로 (`dtype`, `device`) + WebGPU |
| 백그라운드 선로딩 + IndexedDB 캐시 | 체감 대기 ↓, 재방문 0초 |

**최초 채택 조합**: Transformers.js v3 + MobileCLIP(q8) + WebGPU + 선로딩

---

## 5. 구현 (MVP)

- **형태**: 빌드 없는 단일 `index.html` (Transformers.js를 CDN ESM로 로드)
- **기능**: 드래그&드롭 업로드 / 단어 입력(쉼표 다중) / 추천 칩 / 민감도 슬라이더 /
  카테고리별 결과 + 점수 / 미분류 그룹 / 그룹별 ZIP 저장 / 모델 백그라운드 선로딩
- **한국어 처리**: CLIP은 영어 학습 모델 → 내장 사전 `KO_EN`으로 `고양이→cat` 변환(없으면 원문 사용)
- **단일 키워드 정확도**: 숨김 대비 프롬프트(`NEG_PROMPTS`)를 함께 넣어 softmax 대비군 확보

## 6. 실행 검증에서 발견·수정한 버그 ⭐

> 로컬 서버(`python -m http.server`) + 브라우저 미리보기로 실제 이미지(고양이/축구/도시)를 넣어 검증.
> "코드 작성"이 아니라 "돌려보고 확인"에서 다음 문제들이 드러남.

| # | 증상 | 원인 | 조치 |
|---|------|------|------|
| 1 | `WebGPU does not support fp16` | 이 기기 WebGPU가 fp16 미지원 | dtype 폴백 체인 도입 |
| 2 | `[WebGPU] Kernel "[Add]" failed` | WebGPU q8 커널이 런타임에서 깨짐 | 로드 후 1회 추론 검증 + CPU(wasm) 우선 |
| 3 | `Could not locate model.onnx` | MobileCLIP은 vision/text **인코더 분리형** → 고수준 `pipeline()`이 못 읽음 | 두 인코더를 직접 다루는 수동 파이프라인으로 재작성 |
| 4 | `[Add] binary op` (텍스트) | MobileCLIP 텍스트 인코더는 **고정 길이(77)** 입력 필요 | `padding:"max_length", max_length:77` |
| 5 | **고양이를 "사람 96%"로 오분류** | MobileCLIP **텍스트 풀링(EOS 토큰) 불일치** → 텍스트 임베딩 손상 | **검증된 `clip-vit-base-patch32`로 전환** |
| 6 | 업로드 파일이 누적, 지울 방법 없음 | 초기화 기능 부재 | **초기화 버튼 추가** |

### 핵심 의사결정 (#5)
MobileCLIP은 더 작지만, 수동 파이프라인에서 "어느 토큰을 문장 대표 벡터로 쓰는가(풀링)"가
토크나이저와 어긋나 텍스트 임베딩이 망가짐. 블라인드 디버깅보다 **검증된 모델 우선**이 MVP에 맞다고 판단.
→ MobileCLIP은 "추후 과제"로 코드 주석/README에 남김.

## 7. 최종 구성

- **모델**: `Xenova/clip-vit-base-patch32` (q8) — 다운로드 ~150MB(fp32 대비 ¼), 정확도 보장
- **실행**: CPU(wasm) q8 → 실패 시 fp32 자동 폴백 (WebGPU는 불안정해 MVP 제외)
- **라이브러리**: Transformers.js v3 (CDN)

### 최종 검증 결과 (통과)
| 테스트 | 결과 |
|--------|------|
| "고양이" (고양이/축구/도시) | 고양이만 **93%** 정확 분류, 나머지 미분류 |
| 다중 라벨 "고양이,사람,강아지" | 고양이→고양이 **90%** (사람으로 안 샘) |
| 초기화 버튼 | 파일·결과 비움 |
| 민감도 슬라이더 | 임계값 반영 |

## 8. 배운 점 / 메모

- **검증이 핵심**: fp16 미지원, q8 커널 깨짐, MobileCLIP 풀링 오분류 — 전부 *실행해봐야* 드러난 문제였음
- **기기마다 WebGPU 지원이 제각각** → 폴백 체인 + 런타임 검증이 필수
- **MobileCLIP**: 작지만 분리형 인코더 + 풀링 처리가 까다로움. 제대로 쓰려면 EOS 토큰 풀링을 토크나이저에 맞춰야 함
- 고수준 `pipeline()`은 토큰화·풀링·정규화·softmax를 다 처리 → 단일 그래프 모델(clip-vit)엔 가장 안전
